# Бенчмарк document parsing

Сквозной бенчмарк для пайплайнов document-to-Markdown. Версия 1 запускает
текущую цепочку dedoc OCR и `markdown_formatting`. Скоринг потребляет только
предсказанный и эталонный Markdown; от моделей dedoc не зависит.

## Датасет

Датасет по умолчанию —
`research/datasets/ocr_benchmark/manifest.json`. Пути в манифесте заданы
относительно корня репозитория. Preflight-валидация перед запуском OCR
проверяет схему, уникальность кейсов и выборок источников, наличие файлов и
SHA-256 входов.

Нужны заранее подготовленные входы бенчмарка.

## Артефакты последнего прогона

Под `research/benchmarks/document_parsing/output/` хранится только последний
успешный прогон:

```text
manifest.snapshot.json
dataset.digest
cases.jsonl                         # добавляется eval
cases/<case_id>/
  gt.md                           # точный снимок эталона
  gt.canonical.md                 # добавляется eval
  pred.raw.md
  pred.canonical.md                # добавляется eval
  intermediates/
    ocr.json
    formatted.json
```

Пересборка стейджится вне output-директории и публикуется только после
успешного прохождения всех кейсов. При ошибке частичный прогон не заменяет
последний успешный output.

## Команды

Пересобрать артефакты бенчмарка без MLflow:

```bash
python -m research.benchmarks.document_parsing.run
```

Пересобрать, посчитать метрики и залогировать новый MLflow run:

```bash
python -m research.benchmarks.document_parsing.eval.run \
  --source research/benchmarks/document_parsing/output \
  --mode rebuild \
  --name document-parsing-dedoc
```

Пересчитать метрики по существующему output без OCR:

```bash
python -m research.benchmarks.document_parsing.eval.run \
  --source research/benchmarks/document_parsing/output \
  --mode reuse \
  --name document-parsing-dedoc-reuse
```

## Метрики

Все метрики по датасету — macro mean по кейсам. Нет сводного headline-score
и нет предрасчёта агрегации по срезам.

Скоринг всегда идёт по **каноническому hybrid Markdown** (`gt.canonical.md` /
`pred.canonical.md`): Unicode NFKC, `ё`→`е`, снятие inline-форматирования,
HTML-таблицы со native `rowspan`/`colspan` (без flatten merges, без `thead`/`th`),
whitespace→пробел, backend-neutral нормализация из `canonicalize/markdown.py`.
Метрики поэтому измеряют содержимое и структуру после удаления шума оформления,
а не сырой вывод модели.

Per-case скоры лежат в `cases.jsonl`; в MLflow логируются macro mean ниже.

### Текстовый слой (`cer`, `wer`, `token_f1`)

Видимый текст схлопывается в порядке чтения: заголовки, пункты списков,
абзацы и **каждая ячейка таблицы** превращаются в одну нормализованную
plain-text строку.


| Метрика    | Шкала                     | Что измеряет                                                             | Как читать                                                                                                                                                  |
| ---------- | ------------------------- | ------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `cer`      | 0–1, выше = лучше | `max(0, 1 − CER)` по символам                                                  | Чувствительна к OCR-опечаткам, пропущенным символам и мелким расхождениям в написании. Падение на 0.01 часто означает несколько неверных символов на сотню. |
| `wer`      | 0–1, выше = лучше | `max(0, 1 − WER)` по словам                                                    | Сильнее штрафует целиком пропущенные или галлюцинированные слова, чем `cer`. При сыром WER > 1 (много лишних слов) метрика клипится к 0.                     |
| `token_f1` | 0–1, выше = лучше | Пересечение мультимножеств токенов по пробелам                                 | Bag-of-words покрытие лексики без учёта порядка. Высокий `token_f1` при низких `cer`/`wer` обычно значит, что нужные слова есть, но перепутаны или склеены. |


`token_f1` игнорирует порядок токенов и учитывает дубликаты — это сигнал
**покрытия**, а не читаемости. Для экспериментов, ориентированных на
extraction, кейс с низким `token_f1` скорее всего теряет значения, даже если
layout-метрики выглядят нормально.

### Таблицы (`teds`, `teds_s`)

Таблицы сопоставляются **позиционно**: первая предсказанная с первой
эталонной, вторая со второй и т.д. Каждый слот оценивается TEDS по HTML
**со spans** (контракт: origin-only `rowspan`/`colspan`, все ячейки `<td>`).


| Метрика  | Шкала             | Что измеряет                               | Как читать                                                                                                                 |
| -------- | ----------------- | ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| `teds`   | 0–1, выше = лучше | Текст ячеек **и** дерево структуры таблицы | Основной сигнал качества таблиц: неверные заголовки, сдвинутые колонки и плохой текст в ячейках снижают score.             |
| `teds_s` | 0–1, выше = лучше | Только структура (строки, колонки, span)   | Отделяет ошибки вёрстки от OCR-ошибок внутри ячеек. `teds_s` ≫ `teds` → структура в основном верна, текст в ячейках — нет. |


Если на одной стороне таблиц больше, чем на другой, каждый пропущенный или
лишний слот считается за **0**, а score по документу усредняется по
`max(pred_table_count, gt_table_count)`. Одна галлюцинированная или потерянная
таблица тянет весь кейс вниз, даже если остальные таблицы идеальны. Документы
без таблиц по соглашению получают 1.0.

### Структура документа

Структурные метрики симметрично парсят канонический Markdown и намеренно
игнорируют большую часть inline-текста, кроме нормализованных заголовков.


| Метрика                        | Шкала             | Что измеряет                                        | Как читать                                                                                                                                                                                                                                            |
| ------------------------------ | ----------------- | --------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `structural_counts_similarity` | 0–1, выше = лучше | Насколько предсказанные счётчики близки к эталонным | Среднее восьми отношений `min(pred, gt) / max(pred, gt)`: уровни заголовков H1–H6, число table-block и суммарное число строк таблиц (все строки — data rows; header/`thead` в контракте нет). Ловит пропущенные разделы, лишние таблицы и дрейф уровней заголовков.                                                |
| `heading_sequence_f1`          | 0–1, выше = лучше | Упорядоченный outline заголовков                    | Последовательность кортежей `(level, normalized title)` сравнивается через LCS; precision/recall/F1 выводятся из наибольшей общей подпоследовательности. Штрафует перестановку, переименование или пропуск заголовков, даже когда счётчики совпадают. |
| `ast_structure_similarity`     | 0–1, выше = лучше | Дерево блоков документа                             | APTED-similarity по structure-only AST: заголовки, абзацы, списки и скелеты HTML-таблиц со spans (без текста ячеек). Учитывает вложенность и порядок блоков шире, чем одни заголовки.                                                                               |


`heading_sequence_f1` и `ast_structure_similarity` отвечают на разные вопросы.
Heading F1 смотрит на **подписи outline и их порядок**. AST similarity — на
**смесь и вложенность типов блоков** (например, список vs абзац под одним
заголовком). Пайплайн может сохранить правильные заголовки, но проиграть по
AST, превратив списки в абзацы или поставив таблицы не в тот раздел.

### Какую метрику смотреть в эксперименте

Сводный score намеренно не вводится. Берите метрику под гипотезу:

- качество OCR / транскрипции → `cer`, `wer`, `token_f1`
- извлечение таблиц для поиска атрибутов → `teds`; если подозреваете layout,
сравнивайте с `teds_s`
- секционирование, chunking или навигация → `heading_sequence_f1`,
`structural_counts_similarity`
- общая форма Markdown (списки, абзацы, размещение таблиц) →
`ast_structure_similarity`

Поскольку агрегация — macro mean по кейсам, один катастрофический outlier двигает run-level число так же, как множество мелких регрессий. Для интерпретации нужно смотреть per-case строки в `cases.jsonl` и резать позже по лейблам манифеста (`doc_type`, `technical_tags` и т.д.).

## MLflow

Эксперимент: `nsi-attribute-extraction-ocr`.

Run логирует сводные метрики и версии/digest, нужные для идентификации
пайплайна, датасета и scoring contract. Минимальный error-analysis snapshot
содержит манифест, `cases.jsonl`, canonical pred/GT Markdown и текущие dedoc
intermediates. Срезы можно посчитать позже по per-case лейблам и метрикам.

### MLflow Dataset

Adapter логирует ground truth через `mlflow.log_input()` (`context=eval`).
Одна строка на кейс: метаданные манифеста и `reference_markdown` из снимка
`gt.md`. `source` указывает на `research/datasets/ocr_benchmark/manifest.json`;
MLflow сам считает `digest` по содержимому датасета — его видно в run inputs.
Параметр `dataset_digest` в run params остаётся отдельным детерминированным
идентификатором датасета бенчмарка.

Зависимости метрик зафиксированы в `requirements.research.txt`: JiWER для
CER/WER, `table-recognition-metric` для TEDS/TEDS-S и APTED для structure-only
Markdown tree score.
