# E004 - PP-StructureV3: полный CPU baseline для document parsing

## 1. Approach

На полном наборе `ocr_benchmark` сравнивается текущий baseline на dedoc с
внешним документным parser'ом PaddleOCR `PP-StructureV3`. PP-StructureV3
запускается локально на CPU в полной штатной конфигурации для document
parsing: layout, OCR, анализ порядка чтения, таблиц, формул, печатей и
графиков. Для распознавания текста выбирается русско-английская модель
`eslav_PP-OCRv5_mobile_rec`.

Результат PP-StructureV3 фиксируется как самостоятельный snapshot Markdown и
raw-артефактов, после чего оценивается существующим backend-neutral scorer'ом
без изменения рабочего OCR-пайплайна проекта. Сравнение охватывает качество
текста, таблиц и структуры документа, а также практическую стоимость полного
CPU-прогона.

## 2. Expected effect / hypothesis

PP-StructureV3 должен заметно улучшить покрытие текста и таблиц на
image-only scan, повреждённом text layer и сложной вёрстке, где текущий dedoc
даёт пустой или непригодный Markdown. Это должно проявиться прежде всего в
росте `token_f1`, `teds` и `teds_s` без катастрофической потери
`ast_structure_similarity`.

На born-digital PDF эффект может быть нейтральным либо отрицательным из-за
повторного OCR вместо использования native text layer. Если macro-улучшение
незначительно либо достигается ценой неприемлемого времени CPU-инференса,
PP-StructureV3 не оправдывает дальнейшую интеграцию; если улучшение
сконцентрировано на scan-like случаях, следующий шаг — проверка гибридной
маршрутизации страниц.

## 3. Runs and metrics

PP-StructureV3 выполнен полностью на CPU: `PaddleOCR==3.7.0`,
`PP-StructureV3`, `eslav_PP-OCRv5_mobile_rec`, штатные
`batch_size=8` и `PADDLE_PDX_PDF_RENDER_SCALE=2.0`. Все document-parsing
модули оставлены включёнными; `enable_mkldnn=false` — единственная
совместимостная настройка CPU из-за падения Paddle 3.3 в oneDNN на
`PP-DocLayout_plus-L`. Для ограничения пикового RAM документы запускались в
отдельных контейнерах, без изменения их prediction-конфигурации.

## 4. Interpretation

Гипотеза о заметном aggregate-скачке не подтвердилась. PP-StructureV3
существенно лучше восстанавливает часть scan-like и сложных табличных
страниц, но повторный OCR ухудшает уже хороший born-digital text layer и
часто создаёт лишние таблицы там, где GT их не содержит. Итог — падение
`token_f1` почти на 0.10, `teds` на 0.09 и heading F1 на 0.17 относительно
текущего dedoc comparator.

Единственный крупный macro-плюс — `ast_structure_similarity` (+0.089):
layout-дерево и число табличных строк чаще ближе к GT. Часть этого плюса
на `spec-kur0130` следует дисконтировать: старый GT ошибочно ждал три таблицы
вместо одной логической across-pages (reference уже исправлен). Это не компенсирует потерю текста,
outline и времени: полный CPU-прогон занял около 69 минут против
практического сценария, где dedoc использует native text layer.
Эффект PP-StructureV3 выглядит не как глобальная замена, а как кандидат
для строго отобранной fallback-ветки (mixed/scan OCR, catalog-like layout),
а не как «fix across-pages tables».

## 6. Conclusion

**Статус: закрыт.** Полный PP-StructureV3 CPU baseline не даёт основания
заменять текущий `dedoc auto_tabby+linker` глобально. Относительно E002b
падают главные text/table/outline-метрики: `token_f1` −0.096462, `teds`
−0.088292 и heading F1 −0.165278. Даже относительно улучшенного E003-A
результат слабее по этим же направлениям. Рост AST на +0.088789 реален, но
сам по себе не окупает потери содержимого и ~69 минут CPU-инференса на 24
кейса.

Гипотеза частично подтверждена только локально: PP-StructureV3 сильно
восстанавливает часть scan/mixed-layout документов и находит catalog-таблицу,
где `auto_tabby` пасует. Кейс `spec-kur0130` с «across-pages» таблицей
после исправления понимания GT больше не считается выигрышем PP — там
логически права одна таблица dedoc. Одновременно PP ухудшает born-digital
text layer и склонен превращать штампы/рамки в лишние таблицы. Поэтому
наблюдаемый выигрыш не является безопасным, универсальным или экономически
оправданным для текущего пайплайна.

## 7. Decision

- **Не внедрять PP-StructureV3 как глобальную замену dedoc** и не начинать
  сервисную интеграцию/CLI adapter в рабочий OCR pipeline.
- **Сохранить MLflow run `7a686e10ee7a4ab5bd4363fd7003c3db` и его snapshot**
  как воспроизводимый backend-independent reference; он не зависит от
  доступности будущего PP-StructureV3 deployment.
- Перед возвратом к PP-fallback: отдельно проверить дешёвый dedoc-режим
  `pdf_with_text_layer=auto` (без Tabby) на `catalog-belimo` и mixed/space-
  collapse кейсах — см. backlog в E003.
- Если вернуться к PP, сформулировать отдельную дешёвую гипотезу: routing
  только для страниц без пригодного native text layer и с признаками
  mixed/scan / multi-column catalog (не `table_across_pages` по `spec-kur0130`);
  сравнить с E003-A и `auto` по guardrail-метрикам born-digital и подавлять
  штампы/лишние таблицы. Такой тест имеет смысл проводить на GPU или
  выделенном сервере, а не как CPU-default.
