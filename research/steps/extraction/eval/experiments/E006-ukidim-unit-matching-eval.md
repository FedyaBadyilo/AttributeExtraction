# E006 - УКДИМ: eval с учётом единиц измерения

## 1. Approach

Повторная оценка **того же** УКДИМ-объёма и **той же** цепочки извлечения, что в [E003](E003-ukidim-dense-gt.md), с единственным осмысленным изменением — **включённым сопоставлением единиц измерения в eval** (`unit_matching_enabled=true`, флаг `--unit-matching`).

| Параметр | E003 (baseline) | E006 (этот эксперимент) |
| -------- | --------------- | ------------------------ |
| Датасет | УКДИМ: `bak`, `heat_exchanger`, `strainer` | тот же |
| Источник GT / manifest | `research/datasets/processed/ukidim/` | тот же |
| Пайплайн extraction | без изменений (dense 27B grouping, MoE 35B rerank/extraction) | без изменений |
| Eval: value matching | как в E003 | без изменений |
| Eval: unit matching | выключен | **включён** |

**Что меняется в критерии `is_match`.** Для атрибутов с заданными единицами в справочнике класса: при непустых GT и prediction `is_match` требует совпадения **и значения, и единицы** (casefold exact). Для атрибутов без ЕИ и для слотов с пустым GT или пустой prediction правила label-ов не меняются — `unit_match` остаётся `null`.

**Оцениваемый scope по ЕИ.** В УКДИМ GT уже содержит поле `unit` на строках; в справочнике ~265 из 476 extractable-атрибутов помечены как имеющие единицы. Unit matching затрагивает только этот поднабор; остальные слоты сравниваются как в E003.

**Сравнение.** Ориентир — [E003](E003-ukidim-dense-gt.md) (`accuracy` 69.93%, `count` 6010, unit matching выключен). С [E001](E001-baseline.md) и [E005](E005-nsi-processed-dense-grouping.md) (НСИ) **не** сравниваем: другой датасет и другой eval-universe.

Прогон можно выполнить как **re-eval** существующих extraction-выходов E003 (без повторного LLM-пайплайна) — меняется только постобработка и агрегация метрик.

## 2. Expected effect / hypothesis

**H1 — headline `accuracy` не вырастет относительно E003.** Ужесточение критерия на подмножестве слотов с ЕИ может только переводить бывшие `TP` в `FP_2` (значение совпало, единица — нет) или оставлять label без изменений. Рост aggregate `accuracy` маловероятен; ожидается **снижение или плато**, не улучшение.

**H2 — рост `fp2_rate` и/или снижение `tp_rate` на атрибутах с ЕИ.** Основной ожидаемый сдвиг: часть value-only совпадений, ранее считавшихся `TP`, станет `FP_2` из‑за неверной, отсутствующей или не нормализованной `pred_unit` при корректном числовом/строковом значении. Доля затронутых слотов ограничена подмножеством «атрибут с ЕИ + оба значения непусты» (порядка сотен GT-строк с непустым value на unit-атрибутах).

**H3 — слоты без ЕИ и пустые слоты не сдвигаются.** `TN`, `FP_1`, `FN` на атрибутах без `units` и на пустых value должны совпасть с E003; изменения headline-метрик — только за счёт unit-bearing поднабора.

**H4 — диагностическая ценность.** Даже при падении `accuracy` эксперимент полезен: показывает, насколько текущее извлечение **на самом деле** справляется с ЕИ, а не только со значениями. Это входная точка для решения — дорабатывать extraction/prompts под ЕИ или оставлять value-only eval как operational baseline.

**Критерий:** зафиксировать headline-метрики и сдвиг label-ов относительно E003; отдельно оценить вклад unit-mismatch в `FP_2` (error analysis по `rows.jsonl` / `errors/fp_2.json`).

## 4. Interpretation

**Наблюдаемые значения.** После добавления детерминированной нормализации ЕИ и `LLMUnitJudge` (E006b) `accuracy` = **70.32%** — на **0.4 п.п.** выше E003 (69.93%) и на **5.3 п.п.** выше E006a (65.04%). Extraction-выходы те же; меняется только eval-логика unit matching.

**E006a vs E003 — падение было артефактом eval.** E006a (−4.9 п.п. к E003) отражало strict exact-match по ЕИ, а не деградацию извлечения. E006b восстанавливает headline почти до value-only baseline и слегка превосходит его по `tp_count` (+4) при сопоставимом `fn_count` (817 vs 836).

**E006b vs E006a — эффект нормализации и judge.** Переход exact → fixed: **319** слотов `FP_2 → TP`, **2** слота `TP → FP_2`. Из 320 ложных unit-fail при value match в E006a **315** закрыты нормализацией/judge; **5** остаются существенными unit-ошибками (см. §5).

**H1 (accuracy не вырастет) — опровергнут для E006b.** При корректной unit-eval методике headline может быть **на уровне или чуть выше** value-only E003, а не ниже. Исходная гипотеза относилась к ужесточению без нормализации.

**H2 (рост `fp2` из‑за unit) — подтверждается для E006a, снимается для E006b.** E006a: `fp2_count` +332 к E003. E006b: `fp2_count` 363 vs 348 в E003 (+15); прирост объясняется **5 существенными** unit-mismatch при value match + string-judge дрейфом, а не массовой неспособностью извлечь ЕИ.

**H4 (диагностическая ценность) — уточняется.** На slice unit-bearing с непустыми value полный match **83.8%** vs value-only **84.4%** — разрыв **0.6 п.п.**, не ~39 п.п. как в E006a. Реальная проблема извлечения ЕИ узкая: **5** слотов с принципиально неверной или отсутствующей `pred_unit` при верном value.

**Trade-offs.** `unit_judge_call_count` = 18 — LLM вызывается только на остаток после `normalize_unit`; основная масса (801 слот) закрыта детерминированно. Повторный string judge по-прежнему даёт дрейф label-ов относительно E003 (49 переходов `TP → FP_2`, из них только 5 — unit-related); прямое A/B E003 vs E006b по value-only label-ам несимметрично из‑за re-eval.

**Статус интерпретации.** Ясно: strict unit eval завышал ошибки; с normalize + LLM judge unit quality близка к value-only headline. Существенных unit-ошибок extraction мало (5 слотов). Не проверено: стабильность `LLMUnitJudge` на других датасетах; полный разбор 49 `TP→FP_2` vs E003 по string-judge variance.

## 5. Error analysis

Источник: `rows.jsonl` из MLflow runs E006b (`f6591fabe01d4fa1bfafe66419c05ffd`), E006a (`dc138a90086547f588cbb949f1318028`), E003 (`1202f168e07141b5bded7aba8829e7c1`).

### Что было ложным unit-mismatch в E006a

Из **320** слотов E006a с `value_match=true`, `unit_match=false`:

- **315** перешли в `TP` в E006b (`FP_2 → TP`) — **нормализуемые** формы: `°С`/`⁰C`, `мес`/`месяц`, `м^3`/`м³`, `pH`/`ед. pH` и т.п.
- **801** unit-слот закрыт детерминированно (`unit_normalized` 357 + `unit_casefold_exact` 444); **18** дошли до `LLMUnitJudge` (12 приняты, 6 отклонены).

Вывод: доминирующий «unit-mismatch» E006a — **проблема eval-сопоставления**, не extraction.

### Существенные unit-ошибки (E006b)

Критерий: `base_label=FP_2`, `value_match=true`, `unit_match=false` — ошибка **по сути**, не по записи. Таких **5** слотов (0.6% от 820 unit-bearing с непустыми value; 0.08% от `count`):

| # | Атрибут | GT value / unit | Pred value / unit | Класс | Суть |
| - | ------- | --------------- | ----------------- | ----- | ---- |
| 1 | Число полюсов | 4 / `шт.` | 4 / `null` | `bak` | ЕИ не извлечена |
| 2–3 | Гидравлическое сопротивление (греющая сторона) | 48 / `Па` | 48 / `кПа` | `heat_exchanger` | **Масштаб ЕИ**: кПа вместо Па (×1000) |
| 4–5 | Тепловая мощность | 119.3 / `Вт` | 119.3 / `МВт` | `heat_exchanger` | **Масштаб ЕИ**: МВт вместо Вт |

Все 5 отклонены `LLMUnitJudge` (`unit_match_method=unit_llm_judge`, 4 слота) или детерминированно (`unit_mismatch`, 1 слот с `pred_unit=null`). Паттерн: **не перепутанная форма записи, а неверный порядок величины** (Па↔кПа, Вт↔МВт) или полный пропуск ЕИ.

**Примеры**

- `eos=8860323106338646`, Число полюсов: `gt=4 ['шт.']`, `pred=4 [null]` — единственный случай missing `pred_unit` при непустом value.
- `eos=…`, Тепловая мощность: `gt=119.3 ['Вт']`, `pred=119.3 ['МВт']` — число верное, ЕИ завышена на 6 порядков.

### Сравнение с E003

49 переходов `TP → FP_2` между E003 и E006b; из них **только 5** — существенный unit-mismatch при value match. Остальные **44** — `value_match=false` в E006b (string judge / re-eval variance), unit тут ни при чём.

Относительно E003 value-only baseline **5** бывших `TP` корректно понижены до `FP_2` — это реальная плата за честный unit eval, не артефакт.

### Уточнение §4

- Разрыв E003 vs «unit eval» в E006a **не отражал качество extraction**.
- После fix существенных unit-проблем **почти нет**; фокус доработки — **масштаб ЕИ** на атрибутах мощности/давления и редкий пропуск `pred_unit`.
- 2 слота `TP → FP_2` при переходе E006a → E006b — единичный шум judge, не доминирующий паттерн.

### Не разобрано

- Стабильность вердиктов `LLMUnitJudge` (18 вызовов — мало, но без повторного прогона).
- 2 случая `Па` vs `кПа` при **разных** value (48 vs 119.97) в judge-reject — value тоже не совпадает; в «существенные unit-only» не входят.

## 6. Conclusion

Эксперимент показал, что **включение unit matching само по себе не выявляет массовую проблему извлечения ЕИ** на УКДИМ. Первый прогон (E006a, exact match) дал ложную картину: `accuracy` 65.04% (−4.9 п.п. к E003) из‑за нормализуемых расхождений записи (`°С`/`⁰C`, `мес`/`месяц`, `м^3`/`м³` и т.п.), а не из‑за ошибок модели.

После детерминированной нормализации и `LLMUnitJudge` (E006b) headline **70.32%** — на уровне value-only E003 (69.93%); на slice unit-bearing с непустыми value разрыв value vs value+unit всего **0.6 п.п.** (84.4% → 83.8%). Существенных unit-ошибок при верном value — **5** слотов: пропуск `pred_unit` (1×) и неверный **масштаб** ЕИ `Па↔кПа`, `Вт↔МВт` (4×).

H1 (падение accuracy) подтвердилась только для naive exact-match; H2 — для E006a; H4 (диагностическая ценность) — да: эксперимент отделил eval-артефакт от реальных дефектов extraction. Ограничение: re-eval с повторным string judge вносит шум при сравнении с E003; `LLMUnitJudge` проверен на 18 вызовах, без отдельной проверки на НСИ.

## 7. Decision

**Принять E006b** (`normalize_unit` + `LLMUnitJudge` при `--unit-matching`) как рабочую eval-методику для УКДИМ вместо exact-only unit match. Extraction под ЕИ **не блокирует** текущий baseline; точечная доработка имеет смысл только для атрибутов мощности/давления (масштаб SI-префиксов) и редкого пропуска `pred_unit`. Следующий шаг по extraction — не масштабный рефакторинг prompts, а при необходимости отдельный мини-эксперимент на ошибках масштаба; на НСИ — smoke re-eval с той же unit-eval цепочкой.
