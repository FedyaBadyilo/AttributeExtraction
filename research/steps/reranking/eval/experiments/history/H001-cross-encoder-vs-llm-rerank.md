# H001 - Cross-encoder vs LLM rerank

## 1. Approach

После поиска у атрибута остаётся набор объединённых контекстных блоков. Эксперимент сравнивает два способа **переранжирования** перед извлечением:

1. **Cross-encoder** — пара «поисковый запрос – фрагмент», совместная оценка релевантности без LLM на каждый блок.
2. **LLM-реранкер** — модели передаются сведения об атрибуте (тип значения, единицы, подсказка, вариант исполнения при наличии); задача — оценить пригодность блока как **источника значения**, а не только текстовую похожесть.

```mermaid
flowchart TB
  B[Объединённые контекстные блоки] --> CE[Cross-encoder:<br/>similarity query↔chunk]
  B --> LLM[LLM-реранкер:<br/>пригодность как источник значения]
  CE --> W1[Топ-K по CE score]
  LLM --> W2[Топ-K по LLM score]
  W1 --> M[Recall@1 / Recall@5 / ранг]
  W2 --> M
```

Сравнение в ограниченном окне (важно для extraction): Recall@1 / Recall@5, средний ранг, coverage окна.

Проверенные CE-кандидаты: `deepvk/USER-bge-m3`, `bge-reranker-v2-m3-en-ru`.

## 2. Expected effect / hypothesis

**H-CE.** Специализированный cross-encoder поднимет целевые фрагменты выше дешевле и стабильнее, чем LLM.

**H-LLM.** Для ТУ/ТЗ нужна оценка «есть ли связка атрибут↔значение в блоке», а не semantic similarity; LLM с карточкой атрибута улучшит попадание целевого блока в топ-5.

**Критерий.** CE принимаем только при выигрыше по ключевым метрикам (прежде всего Recall@1) без неприемлемого latency. LLM — если R@5 окна растёт при сохранении разумной стоимости.

## 3. Runs and metrics

Исторические результаты серии (без MLflow run ID в этом репозитории).

**Cross-encoder (ранняя проверка):**

| Вариант | Recall@1 | Recall@5 | Средний ранг |
| --- | ---: | ---: | ---: |
| Гибридный поиск без CE | 0.492 | 0.762 | 4.43 |
| CE `deepvk/USER-bge-m3` | 0.254 | 0.571 | 12.22 |
| CE `bge-reranker-v2-m3-en-ru` | 0.413 | 0.794 | 4.03 |

Дополнительно на пакетной проверке CE дал Recall@1 ≈ **0.385** против ≈ **0.507–0.555** у близких гибридных вариантов без CE.

**LLM-переранжирование (окно из 5 блоков):**

| Поверхность | coverage | Recall@1 | Recall@5 | Средний ранг |
| --- | ---: | ---: | ---: | ---: |
| Объединённые блоки до реранка | 0.935 | 0.505 | 0.780 | 3.30 |
| После LLM-реранка, окно из 5 | 0.902 | 0.779 | 0.902 | 1.20 |

Полный список до реранка шире → выше «сырой» coverage; содержательное сравнение — по ограниченному окну (R@5: **0.780 → 0.902**).

## 4. Interpretation

Cross-encoder на этой задаче **не оправдал** ожиданий: ключевой Recall@1 падает, ранг ухудшается или выигрыш по R@5 не компенсирует просадку и рост времени. Направление отклонено.

LLM-реранкер заметно уплотняет контекст: целевой блок чаще первый (R@1 0.505 → 0.779) и почти всегда в пятёрке при среднем ранге ~1.2. Падение coverage полного списка до окна — ожидаемый trade-off отсечения хвоста; внутри окна покрытие целевого источника растёт.

## 5. Error analysis

**Почему CE слабее.** Модель оптимизирует похожесть запроса и текста, тогда как в ТУ много тематически близких, но **непригодных** блоков: соседняя строка таблицы, другой параметр, общее описание без значения, другой вариант исполнения.

**Остаточные ошибки LLM-реранка** на этой итерации: завышенные оценки блокам с похожим параметром / подходящим разделом без явной связки «атрибут – значение» в том же блоке. Ужесточение рубрики оценки — отдельный следующий эксперимент шага.

## 6. Conclusion

Дешёвый CE-реранкер для пилота **не подходит**. LLM-переранжирование лучше отбирает компактный контекст под извлечение, потому что оценивает пригодность источника значения, а не только similarity.

## 7. Decision

**Reject** cross-encoder в базовом алгоритме. **Adopt** LLM-реранкер на объединённых блоках. Следующий эксперимент в этом шаге — группировка контекста близких атрибутов и уточнение рубрики оценки (H002).
