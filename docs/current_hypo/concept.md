# **ОПИСАНИЕ ПРОЦЕССА И ЗАДАЧИ**

## Текущее состояние процесса (AS-IS)

В рамках процесса ведения нормативно-справочной информации в системе после выбора класса МТР эксперт вручную заполняет атрибуты записи МТР, соответствующие выбранному классу. Источником данных служат:

- текстовое наименование МТР (полное описание объекта);
- приложенный файл в произвольном формате (например, PDF), содержащий перечень характеристик МТР.

Эксперт последовательно просматривает наименование и файл характеристик, вручную извлекает значения и переносит их в соответствующие поля карточки МТР. Набор полей определяется выбранным классом и может включать десятки атрибутов.

Ключевые характеристики текущего процесса:

- участники — эксперты системе;
- инструменты — система, приложенные файлы в произвольном формате;
- входные данные — наименование МТР, файл характеристик, перечень атрибутов выбранного класса.

## Проблемы и узкие места

- **Высокая трудоёмкость** ручного извлечения и переноса значений атрибутов из неформализованных документов в карточку МТР.
- **Длительный цикл обработки** заявок при большом объёме номенклатуры МТР и значительном числе атрибутов на каждый класс.
- **Влияние человеческого фактора:** риск пропуска атрибутов, ошибок при переносе значений, неполного или некорректного заполнения карточки МТР.

## Целевое состояние процесса (TO-BE)

Заполнение атрибутов записи МТР выполняется автоматизировано с применением ИИ. На основании текстового наименования МТР, приложенного файла характеристик и актуального перечня атрибутов выбранного класса подсистема ИИ автоматически извлекает значения атрибутов и возвращает предзаполненную карточку МТР в виде структурированного результата. Актуальный перечень атрибутов класса синхронизируется непосредственно из системы.

Роль эксперта смещается от ручного извлечения и переноса значений к проверке и подтверждению предложенного системой результата.

## Идеальный сценарий использования

1.  Конечный пользователь формирует черновик заявки в системе: вводит наименование МТР и прикладывает файл с характеристиками.
2.  Подсистема ИИ проводит автоматическую классификацию и для наиболее вероятного класса – проводит автоматическое извлечение атрибутов
3.  Эксперт выбирает класс МТР (вручную или с помощью подсистемы автоклассификации). Открывается форма с перечнем атрибутов выбранного класса.
4.  В случае изменения автоматически определенного класса, эксперт запускает автоматическое извлечение атрибутов.
5.  Подсистема ИИ анализирует наименование МТР и файл характеристик, сопоставляет данные с перечнем атрибутов выбранного класса и возвращает предзаполненные значения атрибутов с оценкой уверенности в корректности извлечения по каждому из них.
6.  Результат отображается в форме карточки МТР: заполненные атрибуты подсвечиваются с индикацией уверенности. У эксперта есть возможность для каждого значения посмотреть фрагмент исходного документа, из которого извлеклись эти значения.
7.  Эксперт проверяет предложенные значения, при необходимости корректирует вручную и сохраняет карточку.

# **ПИЛОТНЫЙ ПРОЕКТ**

## Цель и задачи пилота

**Цель:** Оценка технологической реализуемости функции автоматического извлечения атрибутов записей МТР и достижимости целевых метрик качества на реальных данных системы.

**Задачи:**

1.  Реализовать функцию автоматического извлечения атрибутов записи МТР по наименованию и приложенному файлу характеристик, включающую:
    a.  извлечение значений атрибутов для заданного класса МТР с оценкой уверенности по каждому атрибуту;
    b.  генерацию сводного отчётного файла, отображающего результаты извлечения с цветовой индикацией уверенности и метриками оценки качества.
2.  Измерить качество автоматического извлечения атрибутов на репрезентативной выборке записей МТР и оценить достижимость целевых метрик.

## Границы и ограничения пилота

**Входит в рамки пилота:**

- Реализация функционального прототипа Подсистемы ИИ для извлечения атрибутов записей МТР по ограниченному набору классов, включающего:
    - приём на вход массив данных по МТР (полное наименование МТР, класс МТР, эталонные значения атрибутов — ground truth) и набора файлов характеристик МТР в формате PDF с текстовым слоем;
    - автоматическое извлечение значений атрибутов для каждой записи на основании наименования МТР, файла характеристик и актуального перечня атрибутов класса из системе;
    - автоматическую генерацию сводного отчётного Excel-файла с результатами извлечения и метриками оценки качества.
- Оценка качества извлечения атрибутов на тестовой выборке с расчётом метрик по методике, описанной в разделе «Критерии успешности и метрики».

**Не входит в рамки пилота:**

- Разработка боевого решения, реализующего целевое состояние в полном объёме.
- Разработка веб-интерфейса и пользовательского UI.
- Полноценная интеграция с системе.
- Обработка файлов PDF, представляющих из себя сканы документов

## Сценарий пилота

1.  **Подготовка входных данных**:
    a.  Выбор двух-трех классов МТР, по которым будет проводиться пилот (например запорная арматура и теплообменные аппараты).
    b.  Набор Excel-файлов с перечнем записей МТР, содержащий для каждой записи: наименование МТР, имя файла характеристик, класс МТР, эталонные значения атрибутов (ground truth);
    c.  Набор Excel-файлов характеристик МТР в формате PDF, соответствующих записям в Excel-файле;
    d.  Набор Excel-файлов с актуальным перечнем классов МТР, атрибутов классов МТР и связи классов и атрибутов из системе.
2.  **Запуск обработки.** Исполнитель запускает процедуру извлечения атрибутов на переданных данных. Система последовательно обрабатывает каждую запись МТР: анализирует наименование и файл характеристик, сопоставляет данные с перечнем атрибутов класса и извлекает значения с оценкой уверенности.
3.  **Формирование сводного отчёта.** По результатам обработки всех записей система автоматически генерирует сводный Excel-файл, содержащий результаты извлечения атрибутов и метрики оценки качества (см. раздел «Критерии успешности и метрики»).
4.  **Анализ результатов.** Эксперт заказчика изучает сводный отчёт, при необходимости вносит замечания по результатам извлечения в отведённую колонку.

## Критерии успешности и метрики

Определите измеримые критерии успешности пилота, целевые показатели и способы их измерения.

**_Бизнес-цель:_** Максимально сократить время экспертов на ручное извлечение атрибутов.

**_Следствия из бизнес-цели:_**

1.  **Максимум** атрибутов должен быть извлечён корректно - эксперт не тратит время на поиск и ввод данных
2.  **Максимум** атрибутов должно быть корректно определено как отсутствующие - эксперт не тратит время на проверку «пустых» значений
3.  **Минимум** ложных извлечений - эксперт не тратит время на исправление ошибок системы
4.  **Высокая уверенность = высокая точность** - эксперт фокусируется только на атрибутах с низкой уверенностью

Решение реализует два направления, которые позволяют эксперту проводить эффективную пост-проверку результатов обработки механизмами ИИ:

- Степень уверенности механизма в достоверности извлеченного значения атрибута, либо в том, что упоминание атрибута отсутствует в тексте наименования МТР и приложенном документе. Извлеченные результаты помечаются одним из двух лэйблов – HC (high confidence, высокая степень уверенности) либо LC (low confidence, низкая степень уверенности)
- Вывод наиболее релевантных искомому атрибуту фрагментов текста в исходных документах – позволяют эксперту моментально оценить результат.

Для данной гипотезы сложность выбора методологии оценки заключается в том, что в промышленной эксплуатации заранее не известен набор атрибутов, который присутствует в обрабатываемом документе. Вследствие этого, классический результат FP распадается на два возможных исхода: 
1) ошибка в значении извлеченного атрибута при его наличии в эталонных данный и 
2) – извлечение значения для атрибута, который отсутствует в эталонных данных. С учетом определения степеней уверенности, полный перечень исходов (меток) применительно к Задаче, представлен в таблице 1.

**_Таблица 1 – метки, связанные с оценкой результата работы ИИ применительно к предложению по цифровизации:_**

|     |     |     |     |
| --- | --- | --- | --- |
| **Результат** | **Описание** | **Ground Truth** | **Предсказание модели** |
| **TP** | Верно найдено и распознано и значение, и единица измерения, если имеется | Давление: 20 Па | Давление: 20 Па |

| **FN** | Пропуск (не смогли извлечь данные атрибута из текста) | Класс безопасности: II | Класс безопасности: пусто |

Для оценки качества извлечения атрибутов вводятся следующие метрики, рассчитываемые автоматически на основании эталонных значений (ground truth) и результатов работы алгоритма.

**1\. Метрика «Доля автоматизации»**

Automation_Rate = (TP + TN) / (TP + TN + FP + FN) × 100%

Показывает: какой процент атрибутов не требует вмешательства эксперта для корректировки результата, полученного по результатам автоматического извлечения.

**2\. Метрика «Время сэкономленное»**

Net_Effect = (_T_old - T_new_) _/ T_old_

Где:

_T_old_ = _T_manual_ × (TP + FP_1 + FP_2 + TN + FN);

_T_manual = 600 секунд – среднее время на обработку атрибута при текущем процессе_

_T_new_ = Σ(TP × _T_check_TP_) + Σ(FP_1 × _T_check_FP_1_) + Σ(FP_2 × _T_check_FP_2_) + Σ(TN_HC × _0_) + Σ(FN_HC × _0_) + Σ(TN_LC × _T_check_TN_) + Σ(FN_LC × _T_fix_FN_)

_Для предлагаемого процесса:_

_T_check_TP_ = 5 секунд — время на проверку корректного значения  
_T_check_FP_1 =_ 10 секунд — время на проверку и обнуление ложного извлечения  
_T_check_FP_2_ = 15 секунд — время на исправление некорректного значения  
_T_fix_FN_ = 20 секунд — время на поиск и внесение пропущенного атрибута  
_T_check_TN_ = 20 секунд — время на проверку корректного отсутствия значения атрибутов

TN_LC и FN_LC – результат работы Решения выдал пустое значение при не высокой степени уверенности

TN_HC и FN_HC – результат работы Решения выдал пустое значение при высокой степени уверенности

При оценке нормативов для проверки должны учитывать следующий момент:  
Для промышленного решения предполагается, что для каждого извлеченного атрибута Эксперту доступны ссылки на место в документе, откуда извлечено значение, а для пустого атрибута - ближайшие фрагменты документа, в которых может присутствовать атрибут, но Решение посчитало, что там его нет.

Пример 1: для атрибутов, которые по факту отсутствуют в ТЗ, но Решение их извлекло, Эксперту при проверке не нужно просматривать весь документ, а только лишь один фрагмент.

Пример 2: Для атрибутов, которые по факту не извлеклись, хотя присутствуют в ТЗ, Эксперт при проверке с большой долей вероятности обнаружит из в предлагаемых фрагментах и не нужно просматривать весь документ.

**3\. Метрика «Верно определенное отсутствие» (Negative Predictive Value)**

**NPV** = TN/ (TN + FN)

Показывает: насколько точно Решение определяет, что атрибут отсутствует в исходном документе и при достижении высоких значений степени уверенности, у Эксперта не будет необходимости делать ручную проверку для результирующих пустых атрибутов.

В ходе пилота эта метрика будет рассматриваться в разрезе степеней уверенности – общая для всех извлечений и для группы извлечений с высокой степенью уверенности **NPV@high_confidence** = TN/ (TN + FN) при условии, что Решение проставило для факта извлечения HC = 1

Для примера, достижение **NPV@high_confidence** значения 99 на практике будет означать, что если исключить проверку человеком результата работы Решения для не определяющих атрибутов с пустым значением, в одном случае из ста: в системе не будет внесено значение атрибута, который на самом деле присутствовал в исходном документе.

По итогам пилота будет создан функциональный прототип подсистемы автоматического извлечения атрибутов записей МТР, включающий:

- **пайплайн оценки качества**, автоматически рассчитывающий метрики качества извлечения на основании эталонных данных (ground truth) и формирующий сводный отчётный Excel-файл.
- **вспомогательный скрипт,** формирующий отчётный Excel-файл.

Взаимодействие с прототипом осуществляется посредством ручного запуска пайплайнов.
